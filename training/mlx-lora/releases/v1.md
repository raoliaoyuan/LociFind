## 4. v0.5 evals 实测对比（with-fallback + hybrid，500 case）

Baseline：parser-only **472** pass / **26** partial / 2 fail（**2026-05-28 含 file_type + screenshot keywords + screenshot extensions 3 轮 parser fix 后**；详 §7 变更历史）。

| 变体 | pass | partial | fail | rescued_to_pass | regressed | variant 命中率 | 字段精确匹配 | p50 fallback (ms) | p95 fallback (ms) | valid_intent 比 |
|---|---|---|---|---|---|---|---|---|---|---|
| Q4_K_M | **480 (96.0%)** | 18 | 2 | 8 | 0 | 99.6% | **96.0%** | 1561 | **1586** | 100% (86/86) |

> 注：Q5_K_M / Q6_K 在 file_type fix 后已实测与 Q4_K_M 完全等同（pass 473），本会话 3 轮 parser fix 后**未重跑** Q5/Q6（parser 修复对所有量化级别同等生效，结论不变）。Q4_K_M 仍是精度饱和 + 最低延迟的 sweet spot，作 v1 默认推理 GGUF。所有变体 p95 远低于 MVP-25 §6.2 阈值 3000 ms。

**BETA-09(a) Windows 跨平台一致性（2026-06-01，Intel Iris Xe Vulkan vs 上表 macOS/Metal）**：同一 Q4_K_M GGUF（sha256 校验一致）跑完整 500 case，**逐项 0pp 差异**——pass 480 / partial 18 / fail 2 / variant 99.6% / 字段 96.0% / fallback 86 / rescued 8 / regressed 0，全部与上表相同。**延迟例外**：弱核显 Vulkan p50/p95 fallback = 19597 / 21858 ms（vs Metal 1565 / 1586 ms），约 13× 慢——硬件等级差距，非正确性问题。详 [docs/reviews/beta-09a-windows-parity.md](../../../docs/reviews/beta-09a-windows-parity.md)。

> **2026-05-28 更新**：本表反映 BETA-08 v1 训练 + BETA-09 量化 + 3 轮 parser fix（file_type / screenshot keywords / screenshot extensions）后的最终实测。累计 pass 460 → **480 (+20 / +4.0pp)**，partial 38 → 18 (-20)，fail 2 不变 / 0 regressed。延迟无变化（parser fix 纯走 parser 路径）。注：第 3 轮 fix（screenshot extensions）让 parser independence ↑（parser-only +1 / hybrid 净增 0；该 case 上轮已被 LoRA 救到 pass，本轮 parser 自己即可 pass，rescued_to_pass 9 → 8 = LoRA 推理压力 ↓）。

## 5. 使用方法

```bash
# 直接加载 GGUF 跑 v0.5 evals (CLI)
LOCIFIND_MODEL_PATH=training/mlx-lora/fused/main-v1-q4_k_m.gguf \
DYLD_LIBRARY_PATH=target/release \
    ./target/release/evals --fixtures v0.5 --with-fallback --hybrid

# 走 ModelDaemon (常驻进程，hybrid 架构)
# - parser 锁 variant
# - 模型只产生 patch 字段（mask-prompt 训练 → valid_intent 比 100%）
# - apply_patch 合并
# 详见 packages/intent-parser/src/hybrid.rs
```

各量化级别选择指南：

- **Q4_K_M (940 MB)**: **默认推荐**。pass 468 / p95 fallback 1592 ms，精度饱和 + 最低延迟，分发场景首选
- **Q5_K_M (1.0 GB)**: 仅在 RAM 富余且 disk size 不敏感时考虑。本 release 实测精度等同 Q4，p95 fallback 2121 ms（+529 ms）
- **Q6_K (1.2 GB)**: 同上。p95 fallback 1952 ms（+360 ms）。"理论最高精度"在本任务不带来 evals 增益

## 6. 已知限制（残留 18 partial buckets，2026-05-28 含 3 轮 parser fix 后）

来自 v0.5 evals 残留：18 partial / 2 fail（Q4_K_M with-fallback hybrid）。按 diff 字段分桶：

| 字段 | 数量 | 典型 diff | 救援前景 |
|---|---|---|---|
| language | 6 | mixed↔zh 检测分歧（"budget" / "几个 G" / "synthetic-place" 等）| 检测器 trade-off 边缘 case |
| artist | 4 | "find songs by synthetic-artist" 未识别 | LoRA 救不动（55 nonempty 中 artist=0），需 Tier 2 augmentation |
| new_name | 3 | "把第5个 rename 为 X" 未提取 | LoRA 救不动（55 nonempty 中 new_name=2），需 Tier 2 augmentation |
| location | 3 | hint 中英形态分歧（"下载" vs "downloads"）| 已收敛到边缘 |
| file_type | 1 | v05-schema-10-010 "找最近一周访问过的 markdown" fixture 漏标 file_type=document（与其他 .pdf/.docx case 不一致）| **fixture artifact**（非 parser bug），需 fixture 修 + dataset versioning |
| title | 1 | media title | 边缘 |
| modified_time | 1 | absolute range vs before type | 边缘 |

**结论**：v1 后再推 pass 数需 **Tier 2 数据 augmentation**（artist + new_name 两个 bucket 各补 ~20 nonempty patch case），单纯加 LoRA 参数 / 调超参 / 提高量化精度都不会再有突破。剩余 1 个 fixture artifact (markdown) 留 backlog（涉及 v0.5-patch dataset 重生 + sha256 锁定问题）。

## 7. 变更历史

**post-release patch 3（2026-05-28，commit `ad162b4`）**：parser 修 screenshot extensions — `media_search.rs` 加 `extract_screenshot_extensions` 函数，仅 `media_type == Screenshot` 时调用；扫 input tokens case-insensitive 命中 jpg/jpeg/png/gif/bmp 即填入 extensions。影响：
- parser-only baseline pass 471→**472 (+1)**
- with-fallback hybrid (Q4_K_M) pass 480→**480 (+0)**（v05-schema-44-045 上轮已被 LoRA 救到 pass，本轮 parser 自己即可 pass，rescued_to_pass 9→8）
- 字段精确匹配率 96.0% 不变
- intent-parser tests 79→**82**（新增 3 test：JPG/PNG case-insensitive / 19-case over-match guard / 单一扩展名）
- 延迟无变化；意义：parser independence ↑ + LoRA 推理压力 ↓

**post-release patch 2（2026-05-28，commit `cf908d9`）**：parser 修 screenshot keywords 3 个 root cause — case-sensitive 比对漏 JPG/PNG / stop_words 不含位置词 / 不含 "一周" 时间词。`media_search.rs` line 449 改 `eq_ignore_ascii_case`；stop_words 加 downloads/desktop/documents/pictures/movies/music + "一周"。影响：
- parser-only baseline pass 465→**471 (+6)**
- with-fallback hybrid (Q4_K_M) pass 473→**480 (+7，LoRA 多救 1 case)**
- 字段精确匹配率 94.6%→**96.0% (+1.4 pp)**
- intent-parser tests 75→79（新增 `tests_screenshot_time_and_stopwords` 4 test）
- 延迟无变化

**post-release patch 1（2026-05-28，commit `4884b04`）**：parser 修 file_type 误识别 bug — 英文 "documents" 仅作位置词，不再触发 file_type=Document。EXTENSION_ALIASES 移除英文 "document"/"documents"，保留中文 "文档"。影响：
- parser-only baseline pass 460→**465 (+5)**
- with-fallback hybrid (Q4_K_M) pass 468→**473 (+5)**
- 字段精确匹配率 93.6%→**94.6% (+1.0 pp)**
- LoRA 救援数 8 不变（救的是不同 case）
- intent-parser tests 72→75（新增 `tests_documents_disambiguation` 3 test 覆盖）
- 延迟无变化（修复纯走 parser 路径）

**v1 训练（2026-05-27）**：mask-prompt + nonempty oversample 8× 攻克 v0 退化解；pass 460→**468 (+8)** / regressed 0 / fallback valid_intent 比 8.3%→**100%**。详 [v1 出场报告](../../../docs/reviews/beta-08-lora-v1.md)

**v0 (2026-05-27)**：管线全通但模型学到 "永远输出 `{}`" 退化解，pass 净增 0，标 not_ready；详 [v0 报告](../../../docs/reviews/beta-08-lora-v0.md)
