# BETA-26 设计：本地语义检索质量探针（BETA-15B embedding 路径去风险）

> 类型：**spike（一周质量探针）**，产出是一个 go/no-go 数字 + 一份方法学，**不是要留的生产代码**。
> 关系：本探针是 [BETA-15B](../../../ROADMAP.md)（"词典升级为 embedding 索引"）embedding 路径的**去风险前置**，专门把 BETA-15B 四道门槛里"门槛①检索相关性怎么测"这一直没说清的方法学做实。
> 边界：守 PROJECT.md「不做完整搜索引擎」+ ROADMAP 第 320 行"不另立全量 embedding 引擎"的既定决策。本探针只测"召回补强"价值，不构建引擎。

## 1. 背景与唯一问题

ROADMAP 第 323 行把 embedding 路径定性为"召回补强实验"，要过四道门槛后才产品化：① BETA-15A 评测集召回率较手维护词典显著提升；② 检索 p95 延迟可接受；③ 常驻额外内存可控；④ 索引体积/后台 CPU 轻量机可承受。

但四道门槛里，门槛①只说"召回率显著提升"，**没说清在什么评测集上、用什么指标、提升多少算 GO**。而 BETA-15A 现有评测集是**同义词召回**专用（query 含可被词典扩展的内容词），**不覆盖"模糊跨格式召回"**——即 query 里**完全不含**目标文件名/正文的精确关键词、只能靠语义命中的那类（同义改述 / 概念主题 / 跨语言 / OCR 内容 / 描述内容而非文件名）。这类正是手维护词典和 FTS5 都搞不定、embedding 唯一可能赢的地方，也是唯一真未知数。

**探针的唯一问题**：在固定真实语料上，**hybrid（FTS5 + 向量）相对 FTS5-only，在"模糊查询"上的检索相关性提升够不够大**？够 → BETA-15B embedding 路径值得继续推进；不够 → 退回纯关键词路径，把资源投别处。

技术可行性已在会话评估中判为 GO（embedding 复用现有 llama.cpp/GGUF 栈、存储用 sqlite-vec 住进现有 SQLite、跨平台已由 BETA-25 趟通、融合/调度/抽取可接 BETA-02/03/04/05/07，无技术拦路虎、无新重依赖）。**唯一该决定 go/no-go 的是质量，质量只能测不能推。**

## 2. 范围护栏（YAGNI——本探针不做什么）

生产管线 / 增量索引 / UI / 跨平台（只在 Mac 上跑，质量与平台无关，跨平台已判 GREEN）/ 后台调度 / 分块调优马拉松 / 多模型横评——**全不做**。一个 embedding 模型、一种融合方法（RRF）、一份冻结语料、一遍跑完。产出是一个数字 + 一句结论。

## 3. 探针五块

### 3.1 检索相关性评测集（新资产，探针核心）

与现有 1000 条 parser-intent byte-equal evals、以及 BETA-15A 同义词召回评测集**都不同**——这里一条 case = "模糊自然语言 query + 在固定语料里**应该**命中哪些文件"。

- 规模：50~100 条，够定向、小到探针周内手标。
- **构造铁律**：query 里**不能含**目标文件名/正文的精确关键词（否则 FTS5 早就赢了，测不出语义价值）。
  - 反例（无效）：query「客服SOP」命中 `客服SOP-v3.docx`。
  - 正例（有效）：query「那份讲我们怎么处理退款的文档」，目标 `客服SOP-v3.docx`、正文写「退货流程」——词面不重叠，只有语义能命中。
- 5 个桶 × ~15 条，压在风险点上：① 同义/改述；② 概念/主题；③ **跨语言**（中文 query → 英文文档，小模型最易跌）；④ **OCR 内容**（query 描述图片里的字，直接测 OCR 噪声风险）；⑤ 描述内容而非文件名。
- 每条 case：`{query, 相关文件ids:[...], 相关度分级 1–3}`（分级用于算 nDCG）。
- 标注纪律：标"按语义**应该**命中什么"，独立于任何检索器**当前**返回什么——复用 BETA-13 已建立的"标应然不标实然"两轴纪律。
- **此评测集是耐久资产**：即便 GO，BETA-15B 调混合权重也要靠它；它补齐了门槛①的方法学缺口。

### 3.2 样本语料（必须真实、必须含噪声）

相关性只在固定语料上有意义。

- 规模：2,000~5,000 文档/chunk，大到检索非平凡、小到几分钟能 embed 完。
- 成分**必须含制造风险的硬货**：zh/en 混排、BETA-03 的 OCR 文本、BETA-02 的 Office/PDF 抽取文本、多格式。
- 来源：用开发机上的真实文件子集（既是标注者又是用户，对"真实个人数据上行不行"最诚实）。**隐私守则**：探针语料/评测集**不进 git**（合成化或本地 gitignore），遵 CONVENTIONS §7"测试样例用合成或公开数据"。
- 冻结快照：三组检索跑同一份语料。

### 3.3 三组检索配置（一次性脚本，不建生产管线）

| 配置 | 内容 | 作用 |
|---|---|---|
| A. FTS5-only | BM25，今天就在跑的 | **基线 / 要打败的线** |
| B. 向量-only | Qwen3-Embedding-0.6B + sqlite-vec，cosine top-k | 诊断纯语义单独多好/多坏 |
| C. Hybrid | A+B 用 RRF 融合 | **真正的候选** |

最小实现：一次性丢弃脚本——llama.cpp 把语料 embed 一遍 → 灌 sqlite-vec → 跑三条检索路径 → 算指标。

候选件（已在会话中评估，spike 期最终确认）：embedding = Qwen3-Embedding-0.6B（同生态、多语 zh+en、GGUF，可对照 bge-m3）；存储 = sqlite-vec（住进现有 SQLite，个人级 <100 万 chunk 暴力扫描足够，不强求 ANN）。

### 3.4 指标

- 主指标：**Recall@10**（对的文件有没有进前 10）+ **nDCG@10**（排得够不够高）。
- **必须分桶报**（按 3.1 的 5 桶）——很可能语义在"跨语言/OCR"碾压、在"精确名"反而添噪；分布比总分更有决策价值。
- 顺手记成本数字（反正都 embed 了，白测验证门槛②③④）：本机 embedding 吞吐、整语料 embed 耗时、int8 量化后磁盘大小、检索 p95。

### 3.5 Kill 标准（**跑之前定死**）

| 结果 | 判定 |
|---|---|
| **GO** | hybrid 在模糊集 Recall@10 ≥ FTS5-only **+15pp**（如 45%→60%+），且精确名子集 hybrid **不低于** FTS5-only（语义没污染已能用的） |
| **NO-GO** | 提升 **< ~8pp**（边际，不值索引成本+复杂度+电池），或 hybrid 明显拖垮精确名查询 |
| **灰区（8–15pp）** | "有戏但小模型是天花板" → 再决定换更大 embedding（bge-m3 / 1.5B）/ 调分块值不值资源，**再**拍板；不直接进 BETA-15B 产品化 |
| **硬 UX kill（独立于质量）** | 16GB 机上真实语料 embed 慢到首索引几小时、且限流压不下去 → 不论质量 NO-GO（对应门槛④） |

- 15pp / 8pp 是起点，由用户按"什么幅度才让用户觉得 10x"最终定。**注意**：对原来返回空的模糊 query，Recall 从 0 拉起来才是"10x 的手感"，别只盯平均分。

## 4. 与现有 task 的关系

- **上游**：BETA-15B（embedding 路径），本探针是它的去风险前置；探针 GO 才进 BETA-15B 产品化（且 BETA-15B 仍受 ROADMAP §B6 升级判定门控）。
- **近亲**：BETA-11B（B3.5 的同义词召回 embedding 升级）与 BETA-15B 是同一技术族（embedding+向量），本探针结论对两者均适用。
- **复用既有评测**：BETA-15A 是同义词召回锚点之一，但本探针**新增**它不覆盖的"模糊/语义/跨语言/OCR"case。
- **复用既有基础设施**（产品化阶段，非探针阶段）：embedding 走 llama.cpp/GGUF（BETA-17/23/25 已就绪）、抽取文本来自 BETA-02/03、索引调度复用 BETA-07、融合排序接 BETA-04/05。

## 5. 方向边界（需用户拍板，探针不预设）

本探针只回答"小本地 embedding 检索质量够不够好"。**若 GO，"做成什么"分两档**，且这是 PROJECT.md 范围决策（CONVENTIONS §9，需用户确认）：

- **保守档（项目现定）**：召回补强实验 → 产品化为 BETA-15B 的召回增强，守"不做完整搜索引擎"边界。
- **进取档（会话中讨论的 Job B 旗舰）**：把"模糊跨格式召回"做成差异化主打能力。这超出现有边界，需用户明确修订 PROJECT.md 范围后才动。

探针本身对两档都是必经第一步，不锁定任何一档。
