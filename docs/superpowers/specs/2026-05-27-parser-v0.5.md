## 1. 背景

MVP-17 fallback 端到端 evals（v0.3 hybrid 架构）揭示：1.5B 模型在当前 fixture 上为最终结果 0 贡献（hybrid pass 108 vs parser-only 109），剩余 fail **100% 是 parser 自己的 variant 错位**。v0.5 全集 500 case 实测：

- **总 fail = 51**（pass 237 / partial 212 / fail 51；variant 命中 89.8% / 字段精确匹配 51.8%）
- **51 fail 中 50 个是过判 FileSearch**（confusion matrix 见 §3）
- 1 个反向（FileSearch → Refine）

修这些纯 parser 工作，0.5-1d，**比 LoRA 微调（BETA-08）性价比高得多**。LoRA 推到 Beta 阶段并入 BETA-08 闭合。

## 2. 目标

| 指标 | 当前 (v0.4) | v0.5 目标 | v0.5 最低 |
|---|---|---|---|
| fail | 51 | ≤ 10 | ≤ 15 |

| 字段精确匹配 | 51.8% | ≥ 53.4% | ≥ 52.4% |

## 3. Confusion matrix（v0.5 全集 500 case）

```
MediaSearch → FileSearch: 23
FileSearch → MediaSearch: 10
Clarify    → FileSearch:  7
Refine     → FileSearch:  6
FileAction → FileSearch:  4
FileSearch → Refine:      1
```

按"根因"重新分桶为 5 个 task：

| Bucket | Count | 代表 query | 根因 |
|---|---|---|---|
| **A1** MediaSearch ← video + 具体 size | 22 | "找下载目录大于 100MB的视频" / "find over 100MB videos in downloads" | v0.4 主动选了"具体 size → file_search"，但 v0.5 全集 fixture 实际契约是反方向 |
| **A2** MediaSearch ← "最重" | 1 | "找最重的视频" | "最重" 漏在 `ABSTRACT_MODIFIERS` 词表 |
| **A3** FileSearch ← video + 时间 + 位置 | 10 | "找上周桌面的视频" / "find videos modified last week in desktop" | v0.4 把 "video + 时间词" 路由到 media_search，但 fixture 这类应走 file_search |
| **B** Clarify ← 太模糊 | 7 | "find recent" / "找 recent 的" / "delete 全部" | parser 缺"只有抽象修饰词无具体目标 → Clarify"触发器 |
| **C** FileAction ← 中文复制/Finder 显示 | 4 | "把这些 pdf 复制到桌面" / "在 Finder 显示第2个" | mixed 语境下中文 "复制" verb 与 "Finder 显示第N个" 模板漏检 |
| **D** Refine | 6 | "清空上一轮位置约束" / "only downloads 里的" / "把这些 pdf 复制到桌面" (39a) | refine clear/delta 模板漏检；部分是 fixture dual-route artifact（同 query 双变体期望） |

剩 1 个 **FileSearch → Refine**（v05-schema-45b-047 "排除压缩包合并后"，方向 expected=FileSearch / actual=Refine）是反向过判，与 39a/39b 同属 fixture dual-route artifact（45a 期望 FileSearch、45b 期望 Refine，parser 只能选一个）。**不在 v0.5 task 范围**，记入已知不可救清单。

## 4. Task 拆分（按 ROI 降序）

### Task 1：反转 v0.4 video size 规则（~32 case）

**范围**：`packages/intent-parser/src/parsers/media_search.rs::has_visual_media_with_abstract_modifier`

**当前规则**（v0.4 落地）：
- `video + ABSTRACT_MODIFIERS (sort/time) → media_search`
- `has_explicit_size_threshold` 出现时 → 走 file_search

**改成**：
- `video + has_explicit_size_threshold → media_search`（救 A1 22 case）
- `video + 抽象 sort 词（最大/largest/最重）→ media_search`（保 v0.4 sort 类）
- `video + 时间词 + 无 size 阈值 → file_search`（救 A3 10 case；从 ABSTRACT_MODIFIERS 拆出 time 词分支）

**实现思路**：
- 把 ABSTRACT_MODIFIERS 拆成 `SIZE_SORT_WORDS`（最大/最小/最重/...）与 `TIME_MODIFIERS`（修改/modified/本周/...）两个词表
- `is_media_query` 改判：
  - 强媒体信号 → media
  - 已知 artist → media
  - `video + has_explicit_size_threshold` → media
  - `video + SIZE_SORT_WORDS` → media
  - `video + 仅 TIME_MODIFIERS（无 size 阈值、无 sort）` → file（即 fall through 不触发 media）

**A2 顺手在 SIZE_SORT_WORDS 加 "最重"**。

**TDD 测试**：
- `video_with_concrete_size_threshold_routes_to_media_search`（用 "找下载目录大于 100MB的视频"）
- `video_with_explicit_size_english_routes_to_media_search`（用 "find over 100MB videos in downloads"）
- `video_with_time_only_routes_to_file_search`（用 "找上周桌面的视频"）
- `video_with_time_english_routes_to_file_search`（用 "find videos modified last week in desktop"）
- `video_with_zhongwen_sort_max_weight_routes_to_media_search`（用 "找最重的视频"）

**预期**：fail−32~33；可能 regression：极少数 v0.4 后靠 time 词通过 media_search 的 case。每个 commit 跑 evals 看 confusion matrix 差。

### Task 2：Clarify 触发器扩展（~7 case）

**范围**：`packages/intent-parser/src/lib.rs` dispatcher 前置

**触发条件**：
- 抽象修饰词（recent / 全部 / all / latest）+ 无具体目标（无 file_type / extensions / keywords / location / artist / time 字段成功提取）→ Clarify with reason="ambiguous_target"
- "delete 全部" / "删 全部" / "delete all" 这类 unsafe verb + 无具体目标 → Clarify with reason="unsafe_action"（与 v0.3 已有 `has_unsafe_delete_signal` 协同，不重复触发）

**注意**：当前已有 `has_unsafe_delete_signal` / `has_ambiguous_*` 系列触发器；新增触发器**优先级最低**，仅当所有其他 variant 判断都得不到具体字段时才生效。

**TDD 测试**：
- `english_find_recent_alone_triggers_clarify`
- `mixed_find_recent_alone_triggers_clarify`
- `mixed_delete_all_alone_triggers_clarify_unsafe`

**预期**：fail−7。Regression 风险：含具体 file_type 的 "find recent files" 不该触发；触发条件要严写"无任何具体字段"。

### Task 3：FileAction 中文复制/Finder 触发（~4 case）

**范围**：`packages/intent-parser/src/parsers/file_action.rs`

**新增**：
- 中文 "把 X 复制到 Y" / "把 X 移到 Y" / "把 X 移动到 Y" → action=copy/move, target_ref 解析 X 引用
- "在 Finder 显示第N个" / "在 Finder 里打开第N个" → action=locate（macOS）/open, target_ref={ "type": "index", "value": N }

**TDD 测试**：
- `chinese_ba_copy_to_routes_to_file_action`
- `chinese_finder_show_nth_routes_to_file_action_locate`

**预期**：fail−4。注意 v05-schema-39b-040 vs 39a-039 是同 query 两个 expected variant（FileAction + Refine），parser 选 FileAction 通过 39b，39a 作为 fixture dual-route artifact 接受为已知失败。

### Task 4：Refine 信号扩展（~5 case，39a 不可救）

**范围**：`packages/intent-parser/src/lib.rs` dispatcher（Refine 触发判断）+ `parsers/refine.rs`

**新增**：
- "清空上一轮 X" / "清空 X 约束" / "清除 X" → Refine.clear=[X]（X 解析为字段名，如"位置/location"）
- "only downloads 里的" / "只看 downloads 的" / "only in X" → Refine.delta.location=X

**TDD 测试**：
- `clear_previous_round_location_routes_to_refine`
- `only_downloads_mixed_routes_to_refine_delta_location`

**预期**：fail−5（不含 39a）。Regression 风险：与现有 refine.rs 规则交叠 — 严格按"显式 clear / only X / 排除 X"短模板触发。

## 6. Tradeoff 政策（沿用 v0.4）

- **允许 partial → fail 局部回归**：只要 aggregate 净正向（fail−N + pass+M 且 M ≥ N）
- **不允许已有 pass → fail/partial 局部回归**：触发就回头加 guard 或缩小触发条件
- 实操：每个 task commit 前必跑 evals 比 baseline，看 per-case diff

## 7. 不在 v0.5 范围

- **fixture dual-route artifact**（同 query 多 expected variant）：parser 物理上无法 100% 通过，报告记入已知失败清单
- **fixture 内部 inconsistent**（同结构不同 expected）：v0.5 按 v0.5 全集统计契约判定，少数 outlier 留 LoRA 处理
- **lexicon 通用结构重构**：v0.5 只补具体词；通用结构动到 v0.6+ 或 Beta
- **hybrid / fallback / prompt 改动**：BETA-08 LoRA 配套统一动
- **GBNF 受限解码**：等 llama-cpp-4 升级
- **evals 工具改动**：当前 baseline + confusion matrix 已足够
