# BETA-15B-5：语义召回可信化（可解释 v1）设计

> 状态：设计已与用户确认（2026-06-20），待用户复审本文档后转 writing-plans。
> 提议 task ID：**BETA-15B-5**（承接 BETA-15B 语义召回旗舰线，可解释/可信化分支）。
> 关联：BETA-15B-1（语义召回纵切 MVP）、BETA-15B-2（后台预热）、BETA-15B-3-A1（相似度下限可调/分数可见）、BETA-20（预览面板）、BETA-26（embedding 去风险探针）。

## 1. 背景与动机

LociFind 的差异化主打能力是**本地、跨语言的语义召回**（"按意思找"）：BETA-15B 已落地文档臂 embedding 索引 + 加权 RRF hybrid 融合，Mac/Windows 真机均验证跨语言命中（中文 query 召回纯英文文档到第 1 名）。

但当前这个能力对用户是**半个黑盒**：结果只显示一个"按意思找到 · 0.XX"徽标，用户无法看到「**为什么这条被召回**」「**doc 里到底哪句话匹配了我的意思**」。差异化能力存在，但**不可感知、不可信**——而"按意思找"这种零词面重叠的召回，恰恰是最需要解释才能让用户信服的。

本项目方向（2026-06-20 用户拍板）：**把差异化做到极致 = 把语义召回的"准"和"可信/可解释"打到顶**。重心为**两者打包、可解释先上**：

- **本 v1（BETA-15B-5）交付可解释/可信**——无评测语料前置、可独立 demo、用户立即可感知。
- **并行启动评测语料隐私决策**——为后续"召回质量调优"（模型天花板/RRF 权重/下限精校）铺路。质量调优本身另起 spec，不在 v1。

## 2. 目标与非目标

### 2.1 v1 目标（in scope）

三个可解释特性，均为**展示层加法**，不改 parser / 索引 / 融合算法：

1. **语义命中段落高亮**（杀手特性）——选中某条语义召回结果时，在预览面板高亮 doc 正文里**与 query 语义最相似的段落**，哪怕零词面重叠。
2. **召回来源 / 原因标注**——每条结果标注命中方式：`纯语义命中（零词面重叠）` / `关键词 + 语义双中` / `仅关键词`。
3. **置信分级 + 下限解释**——把 cosine 分数映射为"强 / 中 / 弱相关"档位；结果区说明"低于 X 的弱相关已隐藏"，解释相似度下限的作用。

### 2.2 非目标（out of scope，明确不做）

- **more-like-this / 语义细化**——独立新交互面，留 v2。
- **召回质量调优**——RRF 权重调优、更大 embedding 模型、相似度下限精校：均**依赖持久化 held-out 评测集**，受 §7 评测语料隐私决策阻塞，另起 spec。
- **索引层改动**——不切块入库、不存 passage 向量、不引入 sqlite-vec（那是 BETA-15B-4）。
- **新增内容臂**（图片视觉语义 / 音频 / OCR 语义）——本 v1 只覆盖现有文档臂。

## 3. 实现路线选型（段落高亮）

三个候选，已与用户确认选 **A**：

| 路线 | 机制 | 取舍 | 结论 |
|---|---|---|---|
| **A：展示时按需算** | 用户选中结果那一刻，只对该 1 篇 doc 切句 → batch embed 段落 → 与缓存 query 向量比 cosine → 高亮 top 段 | 零索引改动、零额外存储；延迟只在显式点击时、只算一篇；段落向量内存即弃 | ✅ **采纳** |
| B：索引时存段落向量 | 切块存 passage 向量进向量库，高亮即时查表 | BETA-26 实测切块成本 7.5× 且对召回不划算；撑大索引；sqlite-vec 未落地 | ❌ 重、提前优化 |
| C：词面代理 | 同义词扩展找词频最密句，不 embed 段落 | 对"零词面重叠的跨语言命中"恰恰失效——正是要解释的场景 | ❌ 自废武功 |

**采纳 A 的核心理由**：demo 真实（真用语义找匹配段，不是词面近似）、零索引/存储改动、延迟可控（只在点击时、单篇）、隐私天然（向量内存即弃不落库）。

## 4. 架构与数据流

复用 BETA-20 预览面板链路，新增一个"展示时解释计算"。

```text
搜索阶段（已有）
  query → parse → hybrid（FTS5 + 语义）→ 结果集
    每条结果已带：cosine 分（语义命中）、FTS 命中标记、来源信息（融合处两路可见）
    query 向量在语义检索时已算 → 缓存复用（关键：解释计算不重算 query 向量）

选中某条语义结果（新增解释计算）
  选中 result（path）
    → 取该 doc 的已索引正文（预览面板已读 index DB 正文，不读原文件）
    → segment(正文) 切句/切段
    → batch embed(段落)  [复用 model-runtime::embed()，模型已由 BETA-15B-2 暖机]
    → rank_passages(query向量, 段落向量[]) → top 段落的 [char_start, char_end] + cosine
    → 返回 ExplainPayload
  前端预览面板把该字符区间用 <mark class="semantic"> 高亮（与 FTS <mark> 视觉区分）
```

### 4.1 模块落点

- **`packages/search-backends/semantic-index`**（或其下新 `explain` 子模块）：纯函数 `segment_passages(text) -> Vec<Passage{start,end,text}>` + `rank_passages(query_vec, passage_vecs) -> Vec<(idx, cosine)>`。纯逻辑、确定性、可单测。
- **`apps/desktop` `search/preview.rs`（或新 `search/explain.rs`）**：新增 `explain_semantic_hit(path, query)` Tauri 命令——薄包装；取索引正文 → 切句 → 调 embedder → rank → 返回 `ExplainPayload { passage_ranges: Vec<(usize,usize)>, scores: Vec<f32> }`。**不调 tracer**。
- **`apps/desktop` 前端**：
  - `PreviewPanel` 接 `explain_semantic_hit`，渲染 `<mark class="semantic">`，悬停显示段落 cosine。
  - 结果行：来源标注徽标（升级现有"按意思找到"）+ 置信档位。
  - 结果区底部：下限说明行。

### 4.2 来源标注判定

来源由 hybrid 融合处的两路命中集判定（融合时 FTS 命中集与语义命中集均可见）：

| 标注 | 判定 |
|---|---|
| `纯语义命中（零词面重叠）` | 在语义命中集、不在 FTS 命中集 |
| `关键词 + 语义双中` | 同时在两个命中集 |
| `仅关键词` | 在 FTS 命中集、不在语义命中集 |

措辞最终以 §9 待确认项为准（用户可微调）。

### 4.3 置信分级

cosine → 档位映射，阈值承接 `SIMILARITY_FLOOR`（BETA-15B-3-A1 已可调、默认 0.30）：

- `强相关`：cosine ≥ 高档阈（建议 ≥ 0.50，待真机校准）
- `中相关`：下限 ≤ cosine < 高档阈
- `弱相关`：< 下限 → **已被下限挡掉、不进结果**（用底部说明行解释，而非展示）

档位阈值定为命名常量，初值保守、留真机校准空间（同 A1 下限的做法）。

## 5. 隐私硬约束（承接 BETA-20 / BETA-21 底线）

- 解释计算**只用已索引正文**（index DB），不读原文件、不触发 OneDrive 水合（沿用 BETA-20 `is_online_only` 在文本路径无关的结论）。
- 高亮段落文本、完整路径、query 向量、段落向量**均不进 trace**——新命令 `explain_semantic_hit` **不调 tracer**，按构造保证（同 BETA-20 `get_preview`、BETA-21/22 命令的做法）。
- 段落向量**只在内存、用完即弃**，不落库（路线 A 的天然好处）。
- 不新增外发、不新增网络调用，守 local-first。

## 6. 延迟与退化

- **Mac（Metal）**：单篇切句 + batch embed 预计百毫秒级，直接出结果。
- **Windows（CPU）**：单篇可能 1–3s → 预览面板显"正在解释命中…"骨架占位，算完替换。复用 BETA-15B-2 已暖机的 embedder（模型已热，省冷加载）。
- **退化路径（必须逐字节等价于现状）**：
  - feature `semantic-recall` 关 / 无模型 → 不显示任何解释 UI。
  - 结果非语义命中（仅 FTS）→ 不触发段落高亮（无 query 向量可比 / 无意义）。
  - embed 失败 / 正文为空 → 静默退化为无高亮的普通预览，不报错阻塞。

## 7. 并行铺路：评测语料隐私决策（v1 不实现，仅登记决策）

质量调优（另起 spec）需要持久化 held-out 评测集来"可度量地变准"。BETA-26 的 68 条 5 桶评测语料是**真机 home 数据、gitignored**（CONVENTIONS §7 隐私底线），不能直接搬进 `packages/evals`。两个选项待用户拍板：

- **① 合成语料**：手造模糊/跨语言相关性 case。可入库、可公开、CI 可跑；但与真实噪声数据分布有失真。
- **② 加密 / gitignored 真实语料**：保真；但需定"加密入库"或"永久 gitignore + 仅本地跑"的约定与工具。

本 v1 **只把此决策登记为并行待办**，不实现评测集。决策定了之后，质量调优 spec（RRF 权重 / 模型天花板 / 下限精校）方可启动。

## 8. 测试策略

- **纯函数单测**（确定性，核心）：
  - `rank_passages`：给定 query 向量 + 段落向量数组 → 返回正确 top 区间与降序分数；边界（空段落集、全部低于下限、NaN/零向量兜底）。
  - `segment_passages`：切句正确性、字符区间不越界、UTF-8 边界安全。
  - 来源标注映射：三种命中集组合 → 正确标注。
  - 置信分级映射：恰在档位阈值边界 / 恰在下限。
- **evals byte-equal 硬门**：本特性是纯展示层加法，不动 parser / 索引 / 融合 → `v0.5 = 473`、`v0.9 = 877`（截至 2026-06-20）parser-only 必须 **0 变化**。改动后逐 case 规范化比对（HashMap 序 + elapsed_ms 非确定，按 id 比 `actual_json`）。
- **feature 关 / 无模型形态**：desktop 测试在该形态下走退化路径，逐字节一致。
- **真机手测（留用户，双平台）**：跨语言 case（中文 query 召回英文 doc）点开预览 → 高亮段是否真命中语义对应句；Windows 验"正在解释命中…"占位 + 暖机后延迟；来源标注/置信档位与直觉一致。
