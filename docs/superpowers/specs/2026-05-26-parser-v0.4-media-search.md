# parser v0.4 攻 MediaSearch + lib.rs 拆分 — Design Spec

> 状态：Draft（brainstorm 已通过，待写 plan）
> 作者：Claude Code (Opus 4.7)
> 日期：2026-05-26
> 前置：parser v0.3 已完成（commit `1868ca9`），evals v0.5 字段精确匹配 47.4% / variant 命中 85.4%
> 关联：[STATUS.md](../../../STATUS.md) 下一步候选 #2 / [ROADMAP §3.2 MVP-25](../../../ROADMAP.md)

## 1. 背景

v0.5 evals 当前分布（500 case）：

| Variant | Total | Pass | Partial | Fail | Pass% |
|---|---|---|---|---|---|
| FileSearch | 200 | 101 | 98 | 1 | 50.5% |
| **MediaSearch** | **100** | **2** | 43 | **55** | **2%** |
| FileAction | 80 | 46 | 30 | 4 | 57.5% |
| Refine | 80 | 74 | 0 | 6 | 92.5% |
| Clarify | 40 | 14 | 19 | 7 | 35% |

**MediaSearch 是真正的最大缺口**（pass 仅 2%），是 v0.5 里唯一的"集体失败"桶。抽 4 条样本看：

- id=13 "找周华健的朋友" → 多余 `extensions: [mp3, flac, ...]`（已 `media_type=audio` 时不应列）
- id=14 "找上个月下载的周华健无损音乐" → 多余 extensions + location + `title="无损音乐"`（quality 词被当成 title）
- id=15 "找我昨天截的付款二维码" → `media_type=audio`（应为 screenshot）
- id=20 "find screenshots from last month" → `keywords=[from, last, month]`（英文 stop words 未过滤）

3-4 个 root cause 集群即可解锁几十条，跟 parser v0.3 同一形态（v0.3 单 task `extensions/.file_type` regex 解锁 43 case）。

`packages/intent-parser/src/lib.rs` 已 1546 行。再叠 v0.4 至少 200-400 行 media 相关，会让后续 v0.5 / fallback 接入更难维护。先拆后改。

## 2. 范围

**在范围内**：

- Task 0：把 `intent-parser/src/lib.rs` 5 个 parser 拆为 `parsers/{file_search,media_search,file_action,refine,clarify}.rs`，行为零变化，43 个现有 test 全过
- Task 1-3：MediaSearch 3 个 root cause（按杠杆排序）
- 收尾：`docs/reviews/parser-v0.4.md` 报告，STATUS / ROADMAP 同步

**不在范围内**：

- FileSearch / FileAction / Clarify 的 root cause（FileSearch 50% 已可接受、Clarify partial 由 Gemini Task A 独立处理）
- 任何 fixture 改动（fixture 是 ground truth，parser 适应 fixture，不反过来）
- llama.cpp / 模型 fallback 启用（由 Codex MVP-17 启动检查独立验证）
- lib.rs 拆分后再加新 helper / 抽象（YAGNI，只拆不重设计）

## 3. 架构变更

### 3.1 lib.rs 拆分后的目录结构

```
packages/intent-parser/src/
  lib.rs                       # 仅留 SearchIntent 类型 re-export、parse() 调度、共享 helper（pub use）
  parsers/
    mod.rs                     # mod 声明
    file_search.rs             # FileSearch parser + 配套单测
    media_search.rs            # MediaSearch parser + 配套单测（v0.4 主战场）
    file_action.rs             # FileAction parser + 配套单测
    refine.rs                  # Refine parser + 配套单测
    clarify.rs                 # Clarify parser + 配套单测
  language.rs                  # 不动
  lexicon.rs                   # 不动（但新增 quality / media_type 词条）
  signals.rs                   # 不动
  prompt.rs                    # 不动
  fallback.rs                  # 不动
```

`lib.rs` 1546 行 → 估计 ~300 行；每个 parser 模块 200-400 行。

### 3.2 模块间依赖

- `parsers/*.rs` import `lexicon::*` / `language::*` / 顶层 SearchIntent 类型
- `parsers/*` 之间不互相 import（解析逻辑独立）
- `lib.rs::parse()` 按 `dispatch_intent_kind()` 调到对应 `parsers::*::parse()`
- 每个 parser 内部保留自己的 `#[cfg(test)] mod tests` —— 测试就近原则

### 3.3 公开 API 不变

外部 caller（`apps/locifind-cli` / `apps/desktop` / `packages/harness` / `packages/evals`）只用 `locifind_intent_parser::{parse, SearchIntent, fallback::resolve_intent, ...}`。`pub use` 链路保持完全兼容。

## 4. Task 编排（三工具并行）

```
Time →
主会话: [Task 0 拆分                ] [Task 1 extensions] [Task 2 media_type] [Task 3 title-quality] [收尾]
Codex:  [MVP-17 fallback 启动检查（独立 worktree，packages/model-runtime + intent-parser/fallback）          ]
Gemini: [Task B 分桶 1-2h] [Task A Clarify 加宽（packages/evals 比较器）                                     ]
```

| 工具 | task | 模块 | 并行性 |
|---|---|---|---|
| **主会话** | Task 0 拆分 + Task 1-3 root cause | `packages/intent-parser/src/parsers/*` + `lexicon.rs` | 独占 intent-parser crate |
| **Codex** | MVP-17 fallback 启动检查 | `packages/model-runtime`（启用 llama-cpp-4 feature + cmake build 验证）+ `packages/intent-parser/src/fallback.rs`（read-only 调用） | 独立 worktree，零冲突 |
| **Gemini Task B（先做）** | MediaSearch 100 case 分桶分析 | 仅读 `packages/evals/fixtures/`，输出 `docs/reviews/parser-v0.4-media-search-buckets.md` | 不动代码，零冲突 |
| **Gemini Task A（后做）** | Clarify partial 加宽 evals 比较器 | `packages/evals/src/lib.rs` 比较器 | 独立 crate，与 intent-parser 解耦 |

### 4.1 派发时序

1. 主会话开 Task 0 拆分 → 同时派 Codex（独立 worktree `codex/mvp-17-fallback-check`）+ Gemini Task B（worktree `gemini/v0.4-buckets`）
2. Gemini Task B 完成（1-2h）→ 主会话用分桶结果定 Task 1 优先 fix 的 fixture 抽样
3. Gemini 接 Task A（同 worktree 或新 worktree `gemini/clarify-comparator`）
4. 收尾顺序：merge Codex MVP-17 报告 → merge Gemini A 比较器变动 → 重跑 v0.5 evals 拿最终 aggregate → 主会话写 `docs/reviews/parser-v0.4.md` → STATUS / ROADMAP 同步 → 一次中文 commit

### 4.2 冲突预防

- Codex worktree 在 `packages/model-runtime`（features 调整）+ `packages/intent-parser/src/fallback.rs`（read-only）。主会话在 `parsers/*` + `lexicon.rs`，**不动 fallback.rs**。
- Gemini Task B **零代码改动**。
- Gemini Task A 在 `packages/evals/src/`，主会话不动 evals crate。
- 若 Gemini A 改了 evals 比较器 → 主会话 v0.4 跑出的 `actual_json` 不变（parser 输出 unchanged），只是 evals 判定更宽松，pass% 数字可能略上升。报告中分别记录"parser v0.4 净提升 vs 比较器加宽净提升"。

### 4.3 三工具调度已知坑（来自 [[three-tool-collab-playbook]]）

- **Codex sandbox 不能写 worktree metadata** → Codex 在 worktree 写完代码 + ci.sh 全过后**无法 git commit**（`.git/worktrees/.../index.lock` 写不进 sandbox），主会话 merge 时需代为 commit。提前在 Codex prompt 写明："ci.sh 全过后停下来，不必尝试 commit；主会话会代为提交。"
- **Gemini 越界倾向**：Gemini 偶尔会改 STATUS.md / ROADMAP.md / Cargo.lock。Gemini prompt 加"**绝对禁止**改这三个文件 + apps/desktop/{App.tsx, main.rs}；只导出新组件 / Tauri command / 评测比较器"。
- **Gemini YOLO 中断可能**：Gemini 偶发 `[ERROR] Invalid stream` 在 commit 前中断，主会话代提交是常态。Gemini prompt 加"完成后输出明确的 `==DONE==` 标记，方便主会话识别"。

## 5. Root cause 攻击顺序（Task 1-3）

按**杠杆**排序：

| Task | Root cause | 假设受影响 case | 一句话改法 |
|---|---|---|---|
| **Task 1** | MediaSearch 多余 extensions | 30-50（凡 media_type=audio/video 都中招） | 当 `media_type ∈ {audio, video, image, screenshot}` 时，不再输出 `extensions` 字段；除非用户显式说"mp3/flac"等具体格式 |
| **Task 2** | media_type 误判（id=15 audio↔screenshot；id=20 stop words 污染 keywords） | 15-25 | 媒体词典优先级 / 截图关键词扩充 / 英文 stop words 过滤 |
| **Task 3** | title vs quality 边界（"无损音乐" 被吞为 title） | 5-15 | quality 词条扩充（无损 / lossless / Hi-Fi / 高清 / HD / 4K），命中后排除 title 提取 |

**真实数字以 Gemini Task B 分桶为准。** 上表是主会话先验。Task 1 完成后立即重跑 v0.5 拿增量。

## 6. TDD 边界

按 v0.3 的成功模式：

- 每个 root cause 在 `parsers/media_search.rs` 内一个独立 `#[cfg(test)] mod tests_<topic>`
- 每个 mod 头加 `#![allow(clippy::unwrap_used, clippy::panic)]`
- 流程：写 failing test → 跑红 → 改 lexicon / regex / parse 逻辑 → 跑绿 → 跑 v0.5 evals 看增量 → commit

Task 0 拆分**没有新 test**，只验证现有 43 个 test 全过（`cargo test -p locifind-intent-parser`）。

## 8. 错误处理

无新增错误路径。本质是 lexicon / regex 修正，已有 `IntentParseError` 链路不变。

## 9. 风险

| 风险 | 缓解 |
|---|---|
| Task 0 拆分 `pub use` 链路出错 → 现有 test 编译失败 | 一目了然，挨个修；TDD 兜底 |
| Task 1 改"不输出 extensions"误伤 file_search（共享 lexicon） | 改动限定在 `parsers/media_search.rs`，file_search 走自己的 parser |
| Gemini Task B 分桶 token 占用大 | prompt 限定"只读 fixture JSON + 输出 ≤ 200 行 Markdown 表" |
| Codex MVP-17 启动检查可能卡 cmake build | 设 30min timeout；Codex 报告 BLOCKED 不影响主会话 v0.4 |
| Gemini Task A 改 evals 比较器后 v0.4 数字与历史报告口径不一致 | 报告中分别记录"parser net" 与 "comparator net"两项增量 |

## 10. 出场标准（v0.4 整体）

完成本会话的 Task 0 + Task 1 + 部分 Task 2/3（按时间）+ Codex MVP-17 报告 + Gemini Task A/B 全部 done 后：

- MediaSearch pass `2 → 30+`
- Aggregate 字段精确匹配 `47.4% → 55%+`
- `docs/reviews/parser-v0.4.md` 落库
- STATUS / ROADMAP 同步
- 一次中文 commit 把所有 v0.4 改动落库

若本会话只完成 Task 0 + Task 1，按 STATUS 收工流程把 Task 2/3 转为下次会话"下一步"。
