# BETA-26 本地语义检索质量探针 — go/no-go 备忘

> 类型：spike 结论备忘（对照 [spec](../superpowers/specs/2026-06-14-beta-26-semantic-retrieval-quality-spike-design.md) §3.5 跑前定死的 kill 标准）。
> 日期：2026-06-15 ｜ 机器：MacBook Pro M5 Pro + 64GB（Metal）｜ 实施：Claude Code (Opus 4.8)，superpowers 全流程。
> 实现分支：`feat-beta-26-semantic-retrieval-spike`（一次性丢弃 crate `packages/spike-retrieval`）。
> 评测集经一轮"加固"：重写词面泄漏 case + 新增 exact-name 守护桶（详见 §4）。

## 结论：**GO**（高置信度）

在真正"词面不重叠"的模糊查询上，**关键词检索（FTS5 trigram BM25）几乎完全失效（Recall@10 = 2.1%），本地小 embedding（Qwen3-Embedding-0.6B）恢复到 88%（hybrid Recall@10 = 88.4%），ΔRecall@10 = +86.3pp**——远超 spec §3.5 设定的 +15pp GO 门槛。且加固桶证明语义**不拖垮精确文件名查询**（exact-name 守护 0/12 回退，反而救回 2 条 FTS5 漏检）。技术可行性此前已判 GREEN；本探针要回答的"质量够不够"现已用数据回答：**够，且差距是质变。**

## 1. 实验配置（spec §3）

- **语料（冻结快照，不入 git）**：整机 home 目录排除构建/缓存目录后分层抽样 **4952 篇**——稀缺格式（office/pdf/html）全收 + md/txt 跨树确定性 stride 补足。格式分布：md 4363 / txt 413 / html 108 / pptx 27 / pdf 21 / docx 14 / xlsx 5 / xls 1。整篇一向量（首 1200 字截断），无分块（spec §2 YAGNI）。
- **评测集（不入 git）**：Claude 读语料起草，用户选"实证复核"（用 FTS5 结果反查泄漏）。**68 条，5 桶**：synonym 14 / concept 14 / crosslang 13 / content-not-name 15 / **exact-name 12（守护桶，加固新增）**。**OCR 桶按计划删去**——本机未接 macOS Vision OCR 引擎（基础设施缺口，留 Mac 会话），无法 OCR 图片。
- **三组检索**：A=FTS5-only（内存 SQLite + trigram，query 按 CJK trigram-OR 构建 MATCH，ORDER BY bm25）；B=向量-only（Qwen3-Embedding-0.6B + 内存暴力 cosine，跳过 sqlite-vec）；C=hybrid（A+B RRF 融合，k=60）。TopK=10，每路 POOL=50。
- **指标**：Recall@10 + nDCG@10，分桶报。

### 纯模糊子集（剔除 FTS5 已能命中的 case，fts_recall < 0.5，共 28 条）

这是探针真正要测的——query 与正文词面不重叠、FTS5 结构性打不中的查询：

| 子集 | n | FTS5 | 向量 | hybrid | Δ(hyb−fts) |
|---|---|---|---|---|---|
| **非泄漏总体** | 28 | **0.021** | 0.908 | **0.884** | **+86.3pp** |
| ├ synonym | 4 | 0.062 | — | 0.875 | — |

**核心发现**：真正模糊的查询上 FTS5 召回近乎 0（2.1%），语义把它拉到 ~88%——正是 spec §3.5 强调的"对原来返回空的模糊 query，Recall 从 0 拉起才是 10x 的手感"。这不是平均分被高分项稀释，而是**质变**。

### exact-name 守护桶（kill 标准"语义不拖垮精确名"——本轮加固才补上）

12 条用真实文件名/标题词的查询，**hybrid vs FTS5 逐条比对：0/12 回退**（hybrid 在每条都 ≥ FTS5）。更意外的是 hybrid **救回 2 条 FTS5 自己漏检的**：e002「中美LLM辩论」(fts 0.33→hyb 1.0)、e010「企业 Agent 落地阻力与治理思路」(fts 0.00→hyb 1.0，FTS5 trigram 没把正确副本排进前 10，语义补上)。**结论：语义召回臂对精确名查询只增不减。**

| 标准 | 阈值 | 实测 | 判定 |
|---|---|---|---|
| GO | hybrid 模糊集 Recall@10 ≥ FTS5 +15pp，且精确名子集不低于 FTS5 | 纯模糊 +86.3pp；exact-name 0/12 回退 | ✅ 两条都满足 |

## 3. 成本数字（顺手白测门槛②③④）

- **embedding 吞吐**：15.8 docs/s（Metal，含已知 per-call context 创建开销）。
- **整语料 embed 耗时**：4952 篇 / 313s（5.2 分钟）。
- **向量体积**：vectors.bin 63.1MB（f32 文本存储；int8 + 二进制可再降数倍）。dim=1024。
- **检索延迟**：暴力 cosine over 4952×1024 + 内存 FTS5，单 query 亚秒级（生产用 sqlite-vec 暴力扫描足够）。

门槛②③④在本机本语料规模上均无压力。

## 4. 关键发现：trigram FTS5 是很强的"同语言"基线

加固过程暴露的重要事实：LociFind 现用的 **trigram FTS5 对中文是强子串匹配基线**。即便把 query 大幅改述、用 trigram 重叠 linter 把字面共享压到很低，BM25 仍常靠**稀有但独特的 3 字片段**把目标顶进前 10（重写后仍有大量同语言 case fts_recall=1.0）。含义：

- 语义检索**可靠、结构性**的胜场是 **crosslang**（中文 query↔英文文档无共享字片段，FTS5 恒为 0）和**词汇真正不相交**的查询（纯模糊子集 28 条，FTS5≈0）。
- 同语言、词面有部分重叠的查询上，trigram FTS5 已经相当能打——这是个**好消息**（现有基线不弱），也校正了"关键词检索在模糊查询上必然差"的直觉。
- 因此语义的产品价值定位应是**补 FTS5 的结构性盲区（跨语言 + 概念跳跃）**，而非全面替代。

## 4.5 分块对照实验（chunk vs 截断，本会话补做）

探针主流程"整篇首 1200 字一向量"是最大捷径。补做分块对照：每篇切 800 字/150 重叠/上限 20 chunk，逐块 embed（37092 chunks），向量检索按 doc 取 chunk 最高分（max-pool）聚合，与截断基线并排比。

| | 向量截断 | 向量分块 | hybrid 截断 | hybrid 分块 |
|---|---|---|---|---|
| 总体 Recall@10 | 0.802 | 0.801 | 0.919 | 0.900 |
| 纯模糊 30 条 | 0.914 | 0.897 | 0.892 | 0.847 |

**结论：分块在本语料上是 wash-to-略负，成本却 7.5×**（vectors 472.8MB vs 63MB，embed 34.7 分钟 vs 5.2 分钟）。原因：① 评测目标的独特信号集中在文档开头，1200 字截断已覆盖，分块无增量；② 长 markdown 干扰项被切到 20 块，多了 20 次 max-pool 蒙高分的机会 → 引入噪声。crosslang 还退 −7.7pp（短文档的干净匹配被长干扰项的最佳 chunk 挤掉）。**对 BETA-15B 的启示：不该默认分块**——截断更便宜且在这类（信号在开头的）数据上不差；只有语料转向"信号埋在长文档深处"时分块才值得。

## 4.6 融合策略调优实验（本会话补做）

针对 caveat ①（纯模糊上 hybrid < 向量-only），复用现有向量（不重 embed）试了几个融合变体：

| 融合变体 | 总体 R@10 | 纯模糊 R@10 | exact-name 守护 | 总体 nDCG@10 |
|---|---|---|---|---|
| RRF k=60（基线） | **0.919** | 0.892 | 1.000 | 0.774 |
| RRF k=30 / k=100 | 0.919 | 0.892 | 1.000 | ≈0.774（k 不敏感）|
| 加权 RRF（向量×2 或 ×3） | 0.891 | **0.925** | 1.000 | 0.758~0.766 |
| query 自适应路由 | =加权 RRF | =加权 RRF | 1.000 | =加权 RRF |
| 向量-only（参照） | 0.802 | 0.914 | 1.000 | 0.679 |

**结论**：① 加权偏向量把纯模糊追回到 0.925（超向量-only 0.914、超基线 0.892），exact-name 守护仍 1.000——但**总体掉 −2.8pp**（修模糊以牺牲词面查询为代价），"模糊 ≥ 向量 + 不掉总体 + 守护不破"三者有真实张力，无单一变体全满足。② **自适应"FTS5 零命中→纯向量"路由形同虚设**——trigram FTS5 几乎从不返回空（总有噪声命中），路由触发器极少激发；启示：生产里该用 **FTS5 置信度阈值**（top BM25 分数）而非"是否为空"做路由。③ RRF k 完全不敏感。④ **所有 margins 都在噪声内（±1 条 case/30）**，且是同一 68 条上的样本内调参——**不能据此生产化，必须在 held-out/扩量评测集上重调**。对 BETA-15B 的启示：融合权重是真实可调的杠杆，但收益小且需独立评测集 + 置信度路由才稳。

## 5. 诚实的 caveats（决策时必须一并看）

1. **纯模糊上 hybrid 略低于向量-only**（0.884 vs 0.908）：FTS5 在这些 case 贡献近零噪声，RRF 掺入反而轻微拖累。hybrid 的价值是**跨查询类型的鲁棒性**（exact-name/同语言靠 FTS5，跨语言/概念靠向量），不是纯模糊峰值。**已做融合调优实验验证（见 §4.6）**：加权偏向量能把纯模糊追回到 0.925，但总体掉 −2.8pp（修模糊牺牲词面），三个目标有真实张力，且 margins 在噪声内——需在 held-out 评测集上调，不能据本轮样本内数字生产化。
2. **无 exact-name 桶的漏洞已在本轮加固补上**（§2 守护结果）——原 spike 此条未测，现已闭合。
3. **语料偏斜**：88% 是代码项目的 markdown 样板，独特可检索目标集中在 office/pdf/中文文档。结论在**更多样化真实个人语料**上的泛化性未测；不过样板正好充当真实噪声干扰项，4952 篇里精确命中少量目标，检索非平凡——这点是加分。
4. **向量只 embed 了首 1200 字**（FTS5 索引全文），有意的不对称、反映现实（关键词索引全文、embedding 受 context 限制）。**已做分块对照实验验证（见 §4.5）**：在本数据上分块并不比截断好（甚至略差），原以为"放开截断/上分块只会更好"的猜测被证伪。
5. **语义并非万能**：`r021`（"一个人的笔记不该绑死在某个工具里、该跟着人走"→云端记忆层 docs）三组全 0——query 用"笔记"、文档讲"记忆/Agent memory"，这层概念跳跃小本地模型没接住。小 embedding 有天花板。
6. **content-not-name 加固说明**：原 17 条多引用正文原话（trigram 直接命中），本轮去掉专有名词/原句重写为语义描述、删 2 条与他桶冗余的，剩 15 条，桶内 Δ 从 +14.2pp 升到 +32.8pp（去泄漏后露出真实模糊价值）。

## 6. 方向选择（spec §5，留用户拍板，探针不预设）

探针只回答"质量够不够好"=**够**。"做成什么"是 PROJECT.md 范围决策（CONVENTIONS §9），两档：

- **保守档（项目现定边界）**：作为 BETA-15B 的"召回补强臂"——在现有 FTS5/多源检索之上加语义召回，守"不做完整搜索引擎"。**风险低、契合现有架构**（embedding 复用 llama.cpp/GGUF，向量住进现有 SQLite via sqlite-vec，融合接 BETA-04/05，调度接 BETA-07）。
- **进取档（会话中讨论的 Job B 旗舰）**：把"模糊跨格式召回"做成差异化主打能力（尤其跨语言——FTS5 结构性的 0，是 Spotlight/Everything/关键词检索都给不了的体验）。**超出现有 PROJECT.md 范围，需用户先修订定位。**

本探针对两档都是必经第一步，不锁定任何一档。**crosslang 的 +100pp 是最硬的差异化论据**。

## 7. 若推进 BETA-15B，探针暴露的待办（非探针范围）

- **融合策略**：已实验（§4.6）——加权偏向量可修纯模糊但损总体、margins 在噪声内；生产应在 held-out 评测集上调权重，且用 **FTS5 置信度阈值**（非"是否为空"）做 query 自适应路由（"FTS5 零命中→向量"在 trigram 下形同虚设）。
- **分块**：已对照实验（§4.5）——本数据上分块 wash-to-略负且成本 7.5×，**不建议默认分块**；仅当语料转向"信号埋在长文档深处"时再评估。
- **embedding 复用 context** 优化吞吐（Task 1 review 已记：单 reusable context + clear_kv_cache_seq，也是生产想要的）。
- 向量存储从文本 f32 换 **sqlite-vec + int8 量化**。
- 评测集（本轮已加固到 68 条 5 桶含守护桶）作为 BETA-15B 调融合权重的**耐久锚点**（spec §3.1）；后续可扩量 + 在更多样语料上复测（caveat 3）。
- 探针 GO 后的**唯一产品化代码遗产** = `model-runtime` 的 `embed()` 路径（已 last-token pooling + L2 归一化，生产质量、双审过）；spike-retrieval 整 crate 可删。
