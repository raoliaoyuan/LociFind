# 企业三场景测试规划

日期：2026-07-03  
场景：律师事务所、内部审计、离职员工归档  
范围：本地索引、语义检索、FTS 混合召回、MCP daemon 调用、权限与取证留痕。

## 1. 总体目标

本测试规划用于验证 LociFind 在企业冷归档场景中能否完成“找得到、找得准、守边界、可追溯”的基础能力。项目定位仍以本地检索底座为准：系统负责索引、权限边界、召回与来源证据，不替代律师判断、审计结论或 HR/IT 管理流程。

核心验收维度：

- 多格式材料可进入索引：DOCX、PPTX、XLSX、PDF、扫描 PDF、JPG、PNG、EML、MD、TXT。
- 真实任务可召回关键证据：条款、日期、金额、人员、项目名、账号、附件来源。
- 相近材料和重复材料可区分：正本、扫描件、转存副本、无关负样本不应混淆。
- 权限边界可验证：不同 collection/root 的可见范围、越权拒绝、审计日志可追踪。
- MCP daemon 可被外部 LLM/agent 稳定调用，返回可回溯的文件路径与片段。

## 2. 律师事务所场景

业务背景：律师需要在案件材料中快速定位合同条款、违约事实、送达证据、庭审记录、和解底线与相似案件，但不同案件之间存在信息墙，不能串案检索。

测试需求：

- 案件集合隔离：同一律师只能查授权案件 collection；查询另一个案件的保密并购摘要应无结果或被权限拒绝。
- 合同条款定位：用“违约金 条款”“延期交付 损失计算”等查询命中合同、律师函和损失测算表。
- 时间线还原：用“交付日期”“送达时间”“庭审记录”命中扫描庭审笔录、邮件和判决书。
- 多版本证据去重：判决书文本层 PDF、扫描 PDF、移动盘副本、低清复扫件都应能关联同一事实，同时保留各自来源。
- 负样本过滤：查询蓝港设备案时，不应把 Northfield 并购摘要作为高相关命中。
- MCP 结果可取证：返回结果应包含路径、文件名、片段、collection 信息，便于写入案件工作底稿。

重点查询样例：

- `违约金 条款`
- `延期交付 损失计算`
- `调解 底线 180万`
- `庭审 送达 日期`
- `设备供应 合同 第十二条`

## 3. 内部审计场景

业务背景：审计人员在采购、付款、验收、供应商往来邮件和举报材料中寻找异常迹象，需要保留取证边界，避免越权读取非审计范围数据。

测试需求：

- 采购闭环检索：合同、报价评分表、验收单、发票、银行回单、审批邮件应可被同一主题串联。
- 异常账户识别：用“收款账户 不一致”“付款账户变更”等查询命中供应商账户异常邮件和银行回单。
- 追溯审批链：用“补审批”“事后审批”“付款暂停”等查询命中邮件链。
- 供应商别名召回：同一供应商的中文名、英文名、简称应能召回相关材料。
- 数值与附件证据：金额、发票号、验收数量差异应能在表格或扫描件材料中定位。
- 职责分离验证：审计 collection 与无关设施采购负样本应保持边界，避免把无关发票混入高相关结果。

重点查询样例：

- `收款账户 不一致`
- `事后审批 采购`
- `验收 数量 短少`
- `供应商 报价 评分`
- `匿名举报 回扣`

## 4. 离职员工场景

业务背景：员工离职后，IT、项目负责人和 HR 需要查找交接清单、系统账号、未关闭缺陷、运维手册、保密协议和离职确认材料，同时避免把其他员工资料误召回。

测试需求：

- 项目资产召回：用项目名、服务名、接口名召回架构说明、值班手册、鉴权说明和截图。
- 未完成事项定位：用“未关闭 bug”“遗留问题”“handover”等查询命中交接邮件和缺陷清单。
- 账号权限盘点：数据库账号、生产权限、双层鉴权等材料应可定位到 CSV/XLSX/DOCX。
- HR 材料隔离：保密协议、离职确认扫描件与技术交接材料同属离职归档，但可按 collection/root 限制可见范围。
- 其他员工负样本：查询李石立相关资料不应高相关命中其他员工 runbook。
- 中英文混合查询：`Lighthouse`、`Kunpeng`、`双层鉴权`、`oncall` 等混合词应稳定召回。

重点查询样例：

- `Lighthouse`
- `双层鉴权`
- `保密协议`
- `数据库账号 权限`
- `未关闭 bug`

## 6. 自动化与人工验证

自动化建议：

- 将 `test-materials/enterprise-scenarios-raw/expected/queries.tsv` 接入 eval runner。
- 每条 query 至少记录 top-k、命中文件、collection、score、是否命中必达 evidence。
- 对负样本设置“不应进入 top-3”或“score 低于阈值”的断言。
- 对扫描 PDF/JPG/PNG 单独标注 OCR 依赖状态，避免与纯文本格式混合误判。

人工验证建议：

- 通过 MCP daemon 用外部 agent 连续提问，检查返回结果是否足够支持下一步分析。
- 抽查每个命中文件能否从路径打开，并确认片段确实来自该文件。
- 用越权 token 或未授权 collection 请求验证拒绝行为。
- 对泛词查询和具体实体查询分别记录表现，区分“召回差”和“意图解析差”。

## 7. 当前已知风险

- PDF/JPG/PNG/OCR 落库链路仍需专项验证；本轮 smoke 已证明文本/邮件/部分原始材料可用，但不能等同于 OCR 全绿。
- 泛词如 `handover`、`项目交接` 存在 degraded 风险，具体项目名和实体词表现更稳定。
- 本轮材料为合成材料，适合回归和开发；BETA-40 严格验收仍需要至少一个真实内网或真实归档目录的 evidence 文档。
