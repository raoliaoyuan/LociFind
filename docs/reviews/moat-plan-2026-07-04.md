## 1. Codex 原方案要点（摘录）

Codex 提出五层护城河 + 优先级序：

1. **场景护城河**：锁死律所 / 审计 / 离职归档三场景，做深不做广。
2. **数据格式护城河**：真实脏材料处理能力（扫描 PDF / CMap / Office / EML / OCR / csv-tsv / 重复件），继续补 `.msg`、附件 evidence、扫描件页码定位、OCR 失败可解释、大目录增量稳定性。
3. **权限与审计护城河**：产品化"AI 可以调用，但权限边界由 LociFind 控制"。
4. **评测护城河**：enterprise eval 22 → 50 → 100 case，fixture + negative + `--require-all` 闸门长期资产化。
5. **MCP 生态护城河**：做外部 AI 的安全检索入口，playbook 产品资产化。

优先级：真实内网证据 → eval 扩容 → 权限/出处闸门 → 失败可解释 UI → 场景 playbook。

一句话定位：**"LociFind 是敏感资料接入 AI 工作流之前，必须先经过的本地语义检索与权限网关。"**

### 2.2 三处结构性修正

**修正一：功能清单 ≠ 护城河。** 本项目自身即反例：BETA-35/36/37 原估时各 1-2 周，实际各 1 天（AI 全程）。AI 编码时代任何单点功能（OCR 管线 / eml 提取 / ACL 模型）竞争对手同样能在几天内复制。壁垒不在"实现了功能"，而在功能背后不可速成的部分：**踩坑 know-how 固化成的评测资产、信任证据、客户侧沉淀、场景纵深**。

**修正二：MCP 是分发渠道，不是护城河。** MCP 协议把服务端商品化——任何人都能写"本地搜索 MCP server"。playbook 该做（获客物料），但投入重心应在接口后面的资产：索引质量、信息墙、审计、可复现评测证据。

**修正三：漏了两层真正难复制的东西。**

- **分发信任**：律所/审计部门不装未签名二进制。Apple Developer / Windows OV·EV 证书（ROADMAP §5 Class A）本身就是企业信任门槛的一部分。
- **客户侧沉淀 = 切换成本**：部署后积累的 per-collection 索引、collection 配置、`audit.jsonl` 审计历史、`content_hash` 去重身份库都留在客户内网。换工具 = 重建索引 + 丢失审计连续性——on-prem 产品最经典的护城河。

## 3. 综合规划（五层，按难复制程度重排）

### 核心命题

> **功能本身守不住（我们 1 天做完原估 2 周的卡，别人也可以）。能守住的是四样不可速成的资产：脏语料 know-how、信任与证据、客户侧沉淀、场景纵深。**

一句话定位采纳 Codex 表述（与 PROJECT.md 一致；是否吸收进 PROJECT.md 定位句待后议）：

> LociFind 是敏感资料接入 AI 工作流之前，必须先经过的本地语义检索与权限网关。

### 第 1 层：证据与信任资产（最难复制，P0）

竞品可复制功能，无法复制"在真实律所卷宗上跑通过、有报告、有客户背书"。合成材料 22/22 全过是底子，BETA-40 验收第二条（真实内网证据）是"看起来能用"到"敢卖"的分界线。

- **找到第一个设计伙伴（灯塔用户）**——哪怕一个律师朋友的一个案子目录、一次离职交接归档。全项目当前最高优先级，从"待用户环境"转为**主动获取**（→ ROADMAP §5 新登记）。
- 证据模板化：每次真实部署产出 `docs/reviews/beta-40-<场景>-evidence.md`，积累成案例库。
- Class A（证书/签名/公证）纳入信任层视角持续推：未签名 = 企业场景零信任。

### 第 2 层：脏语料工程 know-how，以评测资产固化（P1）

中文 CMap panic、WinRT 正斜杠、OCR 污染、2 字 CJK 泛词、csv/tsv 缺口——每个坑都花了真实调试成本，且已固化成回归测试和 `--require-all` 闸门。护城河不是"支持 N 种格式"，而是"每个格式的失败模式都有 fixture + negative + CI 闸门守着"。

- enterprise eval 22 → 50 case（→ **BETA-44**）：优先真实业务复杂度——越权负样本（HR/法务/审计不同 subject）、跨语言别名、近重复干扰、低清复扫件；不为凑数加新格式。
- `.msg`（BETA-37b）维持等真实样本策略；PST 继续不做。
- "踩坑→fixture→闸门"显式化为收工检查（→ CONVENTIONS §3 新增一行）。

### 第 3 层：权限网关与出处闸门（P1-P2）

主打卖点："**AI 可以调用，但权限边界由 LociFind 控制。**"底子已厚：per-collection 物理隔离 index.db、subject 级 audit.jsonl、越权 403 + 防探测同文案、e2e 覆盖。

- **V10-16 先导部分提前**（→ **BETA-43**）：结果强制带出处、禁全文读取时片段级返回、审计导出成合规可读报告（不让客户自己 parse jsonl）。
- V10-16 主卡保留隐私 UI 集成与全量策略收口。

### 第 4 层：场景纵深（P2，随第 1 层真实客户滚动深化）

同意 Codex"锁死三场景、不急扩行业"。补一个抓手：**行业词表 / 别名 / 查询模式**（律所"判决/裁定/庭审笔录"、审计"回单/凭证/供应商别名"）是随真实客户积累的数据资产，合成材料造不出来——playbook 是载体可复制，词表和 query 库不可。

### 第 5 层：MCP = 分发渠道 + 客户侧沉淀 = 切换成本（P3，被动积累）

MCP playbook 定性为**获客物料**："Claude Desktop 5 分钟接上内网卷宗"的体验即传播点。真正锁定来自部署后状态：已建索引、collection 配置、审计历史连续性、去重身份库。行动只一条：**保证这些状态的升级兼容性**（索引 schema 迁移不重建、audit 格式向后兼容），让"续用成本≈0、换走成本=全部重来"。

### 优先级（近期执行序）

1. **P0**：第一个真实部署 + 证据回填（BETA-40 收口；主动找设计伙伴）
2. **P1**：BETA-44 eval 扩到 50 case（负样本/别名/近重复优先）
3. **P1.5**：BETA-33 cycle 9 真机验证（既定节奏）
4. **P2**：BETA-43 出处/权限闸门先导
5. **P3**：Class A 推进（证书/签名，受外部条件限制但持续推）

### 反护城河清单（与 PROJECT.md「不做什么」对齐）

- 不做分析层（已拍板）——"摘要/比对/起草"需求继续引导到 MCP 工作流。
- 不把 MCP 接口打磨当壁垒投入——接口后面的索引、权限、证据才是。
- 不为格式覆盖率做 PST / 重型格式——等真实客户带真实样本来。
- 三场景做到"演示即成交"深度之前，不横向扩第 4、5 个行业。
