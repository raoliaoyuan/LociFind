# ROADMAP v0.1 全局综合审阅报告

> 审阅人：Gemini  
> 日期：2026-05-25  
> 对象：[ROADMAP.md](../ROADMAP.md) v0.1 草稿

## 总体结论

ROADMAP v0.1 在全局逻辑、阶段划分与核心目标上与 [PROJECT.md](../PROJECT.md) 及三份核心计划书保持了高度一致性。任务分解清晰，长周期非代码事项（商标、证书、域名）考虑周全，且正确预留了分发前的公证与签名演练窗口。

作为偏重“全局综合判断”的审阅，我认为该路线图能够支撑 LociFind 从原型到 1.0 的平滑演进。

## 审阅意见分类

### must-fix（v1.0 发布前必改）

*（暂无致命逻辑错误或文档冲突，本项目文档一致性维护良好）*

### should-have（建议在 v1.0 或阶段切换前补齐）

1. **Beta 阶段前增加“法务与安全审查”专项任务**：
   - 引用：[IP 计划书 §13](../docs/LociFind知识产权保护计划书.md)、[风险清单 §12](../docs/LociFind项目注意事项与风险清单.md)。
   - 现状：ROADMAP §5 提到“起草 Privacy Policy / EULA”，但在 B 阶段任务清单中未明确体现针对 Beta 外发版的“正式法务审查”。
   - 建议：在 B 阶段（如 BETA-10 之前）增加一个显式任务：`BETA-00 法务与安全最终审查（EULA / 隐私条款 / 第三方授权核查）`。

2. **长周期事项补充“全球商标评估”**：
   - 引用：[IP 计划书 §4.3](../docs/LociFind知识产权保护计划书.md)。
   - 现状：ROADMAP §5 仅列出中美商标申请。
   - 建议：在 MVP 启动前增加一个子任务：`评估欧盟/英国/新加坡/香港等第二梯队地区的商标申请需求`。

3. **明确性能基准测试的硬件画像**：
   - 引用：[项目计划书 §16.2](../docs/本地个人搜索Agent项目计划书.md)。
   - 现状：ROADMAP §6.2 提到“16GB 机器上流畅运行”。
   - 建议：在 M 阶段出场指标中明确测试画像（如：8核 CPU / 16GB RAM / 512GB SSD），以确保“轻量可用”原则的可量化验证。

### nice-to-have（可选优化）

4. **本地活动洞察（V10-02）的“渐进式”提前**：
   - 现状：目前完全放在 1.0。
   - 建议：如果 Beta 阶段进度领先，可考虑将“最近文件统计”（不含主题摘要）作为 Beta 的实验性功能放出。这能显著增强 Beta 版的“Agent 感”。

5. **风险地图增加“隐私政策变更”项**：
   - 现状：目前风险地图侧重技术与 IP。
   - 建议：增加“应用商店/系统平台隐私政策收紧”风险（如 Apple 对隐私清单的新要求），确保 Beta 阶段适配工作不被突发合规要求卡住。

### out-of-scope（明确不做）

- **云端同步功能**：确认 ROADMAP 全程未出现云端同步任务，符合“本地优先”原则。
- **第三方 Plugin 商店**：V 阶段仅包含 Plugin SDK，不应包含维护商店的运维任务。

## 对 ROADMAP 附录 A 给 Gemini 的 6 条问题答复

**1. 与 PROJECT.md / 三份计划书 / 风险清单是否一致？**
> **结论：一致。** 
> ROADMAP 准确转化了“技术原型 -> MVP -> Beta -> 1.0”的演进路径。
> 关键点（Everything 可插拔、Spotlight 优先、Tauri 框架、MLX 训练）在 ROADMAP 各阶段任务中均有体现。

**2. §5 长周期事项是否齐全？还有什么没列？**
> **结论：基本齐全。** 
> 已包含域名、商标（中美）、Apple/Windows 证书、隐私政策、公证演练。
> **补充建议**：补充“第三方 Notices 台账维护”（见 §5 已列，但建议在 PROTO-01 开始时强制执行）。

**3. §4 里程碑（M0-M4）是否覆盖了关键节点？**
> **结论：覆盖充分。** 
> M0（设计）、M1（原型）、M2（MVP 内测）、M3（Beta 公测）、M4（1.0 发布）的逻辑链条完整，与产品发布节奏匹配。

**4. V10-02 本地活动洞察模块的位置 / 时长 / 依赖是否合理？是否提前？**
> **结论：位置合理。** 
> 该模块涉及持续的本地活动记录与隐私敏感度，放在 1.0 作为“高价值增强”非常稳健。
> **建议**：如需增强 Beta 吸引力，可将简单的“最近使用文件”逻辑下沉到 Beta 的 Result Normalizer 增强中。

**5. §7 风险地图与风险清单是否有遗漏 / 重复？**
> **结论：匹配度高。** 
> ROADMAP 提取了风险清单中的核心工程风险（模型慢、Spotlight 权限、跨平台复杂度）。
> **补充建议**：可追加“模型授权在不同司法辖区的商用限制”作为低概率/中影响风险。

**6. ROADMAP / STATUS / PROJECT / CONVENTIONS 四份文件的分工是否清楚？**
> **结论：非常清楚。** 
> §0 的表格清晰定义了“目标 (PROJECT) / 地图 (ROADMAP) / 位置 (STATUS) / 规则 (CONVENTIONS)”的四合一关系。
